## Overview

按 v1.0.0 任务清单，本期 5 sprint 粗粒度 phase-2 翻译。本文件给**每个 sprint 的翻译约定、模块分解、关键决策、测试设计、风险**。

**v0.6 阶段已有的相关基础**（sprint 1+ 直接复用）：
- `[*]T` 切片 codegen（6A）
- 模块命名空间 `mod::fn()`（6B）

- NODE_ADDR_OF / NODE_DEREF / NODE_FIELD 修复（6D）
- extern fn 不被 mangle（6D）
- Stage 0 arena.jhyy 翻译（6E）— **已删，本期重新翻译**
- mcp-jhyy 服务（v0.5）

## 翻译约定（每个 sprint 都用）

### 类型映射（C → jhyy）

| C 类型 | jhyy 类型 | 说明 |
|---|---|---|
| `int` / `i32` | `i32` | |
| `long long` / `i64` | `i64` | |
| `size_t` | `i64` | jhyy 不区分 size_t，统一 i64 |
| `bool` (`<stdbool.h>`) | `bool` | jhyy 原生 |
| `void*` | `*u8` | 不透明指针 |
| `T*` (typed) | `*T` | |
| `char*` (字符串) | `[*]u8` | 用 slice，NUL 终止 |
| `char[N]` (小缓冲) | `[u8; N]` | 数组 |
| `struct Foo` | `type Foo = struct { ... }` | |
| `union { i32 k; void* p; }` | `i32` + `*u8` 平行 + as cast | tagged union 模式 |
| `enum E { A, B }` | `i32` 常量 | jhyy 无 enum 原生 |
| `const char*` literal | `"..."` | |
| `T**` (array of ptr) | `[*]*T` | |

### 字符串处理

- C `strlen` / `strcmp` / `strncmp` → jhyy 用 `[*]u8` slice + 自己实现（`util.jhyy` sprint 1 提供 `slice_eq`、`slice_len`）
- C `strdup` → `arena_strdup`（v0.6 stage 0 已有）
- C `snprintf` / `fprintf` → FFI `fprintf`（变参 FFI v0.6 未做，先用 `extern fn puts` + 字符串拼）

### Arena 模式

- C `arena_alloc(a, sizeof(T))` → `arena_alloc(a, size as i64)`（**jhyy 无 `size_of::<T>()` 内置**，sprint 2 types.jhyy 翻译时用 ABI sizeof 表硬编码 + match 递归实现 `type_size(*Type) -> i64`）
- C `arena_calloc` → `arena_calloc`（sprint 1 提供）
- 数组 C `T *arr = arena_alloc(a, n * sizeof(T))` → `let arr = arena_alloc(a, (n * size) as i64)`（调用方**自己算字节数**，jhyy 无泛型所以无 `alloc_slice::<T>`）

### 错误传播

- C `bool func(...)` 返回成功失败 → jhyy 同
- C `int func(...)` 0=成功 -1=失败 → jhyy `i32`，约定 0=成功
- C `T* func(...)` NULL=失败 → jhyy `*T`，约定 NULL ptr=失败
- 不引入 `Result<T, E>` 模式（**jhyy 无泛型**，phase-3 sprint 3b 才考虑）

### FFI 声明位置

- 所有 `extern fn` 集中在 `compiler/src0/ffi.jhyy`
- sprint 1 第一步：建 ffi.jhyy，列 libc 必需函数（malloc, free, memcpy, memset, fprintf, fopen, fread, fclose, strlen, strcmp）

### 命名约定

- C `snake_case` → jhyy `snake_case`（不变）
- 翻译模块 = C 文件名（`arena.c` → `arena.jhyy`、`parser.c` → `parser.jhyy` 等）
- 新模块（**不在 C 端**）：`util.jhyy` / `ffi.jhyy` — sprint 1 新建
- FFI 函数名前加 `libc_`（如 `libc_malloc`）— 跟 jhyy 自己函数区分

### C ↔ jhyy 互调（sprint 1 M1 起）

- C 编译器调 jhyy 写的 lib：C 主程序 link jhyy 编译出的 .o + 写 C 头文件声明
- 当前 sprint 1 M1 选小范围：让 `compiler/src/arena.c` **不重写**，但加一个 `compiler/src/arena_jhyy_bridge.c` 调 jhyy arena 的 FFI 函数 — **这是 M1 的验证载体**

### 1.2 util.jhyy 翻译要点（**抽取型**）

**StringBuilder**（从 codegen.c 抽 API，**新建类型**）：
- C 现状：**codegen.c 6 处 `snprintf`** 调用（行 223/226/228/231 浮点 nan/inf 转换 + 行 428/482 名称 mangle），每次都重新算 buffer 大小
- jhyy 等价：`type StringBuilder = struct { buf: [*]u8, len: i64, cap: i64 }`
- 关键操作：`sb_init` / `sb_append` / `sb_append_cstr` / `sb_appendf`（format string + 固定参数拼接，**v1.0 用 libc_sprintf 包装**）/ `sb_cstr` / `sb_reset` / `sb_free`
- arena 分配 buf，扩容时 libc_realloc
- **sprint 1 末 C 端 codegen.c 不重写**，仅 sprint 4 翻译 codegen.jhyy 时再删 sprintf 调用

**HashMap**（**翻译自 symtab.c**，不是抽取）：
- C 现状：symtab.c 共 6 个函数 — `hash_string`(static, FNV-1a 64-bit) / `symtab_grow`(static) / `symtab_new` / `symtab_insert` / `symtab_insert_sym` / `symtab_lookup` / `symtab_lookup_local` / `symtab_lookup_one`(static)
- jhyy 等价：`type HashMap = struct { buckets: [*]*u8, nbuckets: i64, count: i64, arena: *Arena }`
- 用 `*u8` + 显式 cast（v0.6 stage 0 arena.jhyy 已用此模式）
- FNV-1a hash（v0.6 sprint 6C 加了 `as usize` 支持 hash 算术）
- **sprint 2 翻译 symtab.jhyy 时直接复用 util.jhyy 的 HashMap 替代 C 端实现**

**Diagnostic**（**翻译自 sema.c**，不是抽取）：
- C 现状：sema.c **只有 1 个 error 函数 + 1 个 vfprintf**（行 8-13：`void error(SourceLoc loc, const char *fmt, ...)` + `vfprintf(stderr, fmt, ap)`）；parser.c 另有 3 处直接 fprintf（行 17/178/555）也走同一格式 `filename:line:col: error: ...`
- jhyy 等价：`fn error(loc: SourceLoc, msg: [*]u8)`（**v1.0 不做变参**，先固定 msg）
- `SourceLoc` 是简单 struct `{ filename: [*]u8, line: i32, col: i32 }`（v0.6 已用）
- 输出到 stderr：FFI `libc_fputs(msg, stderr)`（不变参，避开 variadic 限制）
- **变参处理**：jhyy 端调用 error 前**先用 StringBuilder 拼好完整消息**（如 `sb_appendf(&sb, "expected %s, got %s", ...)` 拼成单个字符串再传给 error），彻底绕开 JHYY 端 variadic

extern fn fopen(path: [*]u8, mode: [*]u8) -> *u8;
extern fn fread(buf: *u8, size: i64, count: i64, fp: *u8) -> i64;

// ─── 内存 ───
extern fn malloc(size: i64) -> *u8;

extern fn memcpy(dst: *u8, src: *u8, n: i64) -> *u8;
extern fn memset(s: *u8, c: i32, n: i64) -> *u8;

extern fn strcmp(a: [*]u8, b: [*]u8) -> i32;
extern fn snprintf(buf: *u8, n: i64, fmt: [*]u8, ...) -> i32;

### 2.2 ast.jhyy 翻译要点

**Tagged union 模式**：
- C `typedef struct Node { NodeKind kind; ... payload union; } Node;`
- jhyy 等价：`type Node = struct { kind: i32, payload: *u8 }`（payload 是 untyped，访问时 `as *NodeBinary`）
- 跟 v0.6 stage 0 arena.jhyy 的 `ArenaBlock` 模式一致
- 每种 NodeKind 配 payload struct（`NodeBinary` / `NodeCall` / `NodeIf` / ...）

**构造函数**：
- C `static Node *new_node(Arena *a, NodeKind k)` → jhyy `fn new_node(a: *Arena, k: i32) -> *Node`
- payload 构造用 `arena_alloc` + field-by-field 设置

### 2.3 types.jhyy 翻译要点

**递归类型**：
- C `struct Type { TypeKind kind; Type *elem; ... }` — 指针递归
- jhyy 等价：`type Type = struct { kind: i32, ... }` + 所有递归字段用 `*Type`
- 关键 API：`type_primitive` / `type_pointer` / `type_array` / `type_struct` / ... / `type_size` / `type_align` / `type_eq` / `type_to_string`

**type_size / type_align 实现（sprint 2 启动预检发现关键事实）**：
- **jhyy 无 `size_of::<T>()` 内置**（grep `docs/abis/` + `tests/examples/` 全无；v1.0 详细方案之前的"v0.6 sprint 6A 提到"是错的）
- codegen.c C 端走 `type_size()` 函数（types.c 里），本质是**按 TypeKind switch + 递归**
- jhyy 翻译：**硬编码 ABI v1.0.0 sizeof 表 + match 递归**

```jhyy
// 基础类型硬编码（来自 docs/abis/jhyy-abi-v1.0.0.md § 类型布局）
fn primitive_size(kind: i32) -> i64 {
    match kind {
        KIND_INT | KIND_BOOL => 4,
        KIND_LONG => 8,
        KIND_PTR => 8,
        KIND_FLOAT => 4,
        KIND_DOUBLE => 8,
        _ => 8  // 默认 8 字节
    }
}

// 复合类型递归
fn type_size(t: *Type) -> i64 {
    match t.kind {
        KIND_PRIMITIVE => primitive_size(t.prim),
        KIND_POINTER => 8,
        KIND_SLICE => 16,            // {ptr, len}
        KIND_ARRAY => t.array_count * type_size(t.array_elem),
        KIND_STRUCT => ...           // 累加 field_size
        KIND_FN => 8,                // 函数指针
        _ => 8
    }
}
```

参考 `docs/abis/jhyy-abi-v1.0.0.md` 完整 sizeof 表（行 33-176 含 struct 偏移 / 切片 / 数组 / 函数指针 / Unit 类型）。

**type_align 同理**：原始类型 alignof == sizeof（bool 除外，bool=1），切片 16 = max(ptr, len)，struct = max(field align)。

### 2.4 symtab.jhyy 翻译要点

- 复用 util.jhyy 的 HashMap
- scope 链：`type SymTable = struct { map: HashMap, parent: *SymTable, depth: i32 }`
- `symtab_lookup` 递归 parent，`symtab_lookup_local` 仅当前层

### 2.5 风险

- **AST tagged union 是 v1.0 最大模式决策** — sprint 2 第一周先做小范围（5 个 NodeKind 试），通了再展开

- `size_of` / `align_of` 内置是否实现：sprint 2 第 1-2 天**先验证**（读 codegen.c 看怎么 emit alloc 大小）

### 2.6 验证

1. `jhyy (C 版) 编译 src0/{ast,types,symtab}.jhyy` 全成功
2. ast 单元测试：5 个 NodeKind（INT/BINARY/CALL/IF/LET）的构造 + 字段访问
3. types 单元测试：`type_size(i32) == 4` / `type_size(*u8) == 8` / `type_size([u8; 16]) == 16` / `type_eq(int, int) == true` / `type_eq(int, i64) == false`
4. symtab 单元测试：插入 + 查找 + 遮蔽 + 跨 scope 查找
5. **M2**: C 编译器调 FFI 用 jhyy ast/types/symtab — 类似 M1，薄包装 + .o
6. `regress.py 43/46 全过`

## Sprint 3: 流水线 (lexer + parser + sema)

### 3.1 翻译目标

| C 文件 | jhyy 文件 | 复杂度 |
|---|---|---|
| `compiler/src/lexer.h/c` | `compiler/src0/lexer.jhyy` | 中（30+ token type，状态机） |
| `compiler/src/parser.h/c` (982 行) | `compiler/src0/parser.jhyy` | **高（拆 3 commit）** |
| `compiler/src/sema.h/c` | `compiler/src0/sema.jhyy` | 中-高（类型推断 + 检查） |

### 3.2 parser 拆分（sprint 内部多 commit）

sprint 3 **必须**拆 2-3 个 commit：
- **commit 1**: parse_decl（顶层 fn / type / extern / import 声明）
- **commit 2**: parse_stmt（let / if / while / for / match / return / block）
- **commit 3**: parse_expr（Pratt 解析：前缀/中缀/后缀）

每个 commit 末跑 `jhyy 编 parser.jhyy` + `regress.py 43/46`。

### 3.3 lexer.jhyy 翻译要点

- C `TokenKind` enum → jhyy i32 常量（参照 ast.jhyy 模式）
- 状态机用 while + 状态变量（不要 match-jhyy 嵌套过深）
- `lexer_peek` 不消耗 token — jhyy 等价：返回 `*Token`（arena 分配的引用）
- 关键字识别：哈希表查 `keyword_kind(name)`，复用 util.jhyy HashMap

### 3.4 parser.jhyy 翻译要点

**Pratt 解析**：
- C 用 `static Node *parse_expr(Parser *p, Precedence min_prec)`
- jhyy 等价：`fn parse_expr(p: *Parser, min_prec: i32) -> *Node`
- 关键：parse_rule 表（`ParseRule[]`）→ jhyy 用 array of struct + 索引

**parse_match / parse_struct_lit**：
- match arms 用 linked list（`NodeMatchArm *next`）→ jhyy 用 `*NodeMatchArm` 链
- struct lit fields 用 `[*]NodeStructField`（v0.6 切片支持）

**翻译工作量评估**：
- 982 行 C → ~1500 行 jhyy（jhyy 字符串处理 + 类型注解比 C 啰嗦）
- 拆分后：每 commit ~300-500 行 jhyy

### 3.5 sema.jhyy 翻译要点

- 两遍遍历：Pass 1 收集顶层声明，Pass 2 分析函数体
- 类型推断用 util.jhyy 哈希表 + 递归 unifier
- `match` 模式用 jhyy 的 match 表达式（C 的 switch 改写）

### 3.6 风险

- **parser 982 行 = sprint 3 最大单模块**（注：codegen.c 1552 更大，但属于 sprint 4）。如果拆分后单 commit 仍太大，**降级**为 sprint 3a / 3b 拆两个 sprint
- Pratt 解析的 jhyy 表达：函数指针 (`Node *(*infix)(Parser *, Node *)`) 在 jhyy 无 — 用 enum kind + match 替代
- 类型推断的 jhyy 递归：深度可能很深的递归，jhyy 无 TCO 优化 — sprint 3 末验证不爆栈

### 3.7 验证

1. `jhyy (C 版) 编译 src0/{lexer,parser,sema}.jhyy` 全成功
2. lexer 单元测试：每个 token type 至少 1 个测试
3. parser 单元测试：每个 parse_func 至少 1 个测试（parse_decl / parse_stmt / parse_expr 分开测）
4. sema 单元测试：类型推断（let 推断 / 显式标注）+ 错误检测（未定义变量 / 类型不匹配）
5. **M3**: C 编译器走完整流水线 — `jhyy (C 版) 编译 src0/{lexer,parser,sema}.jhyy`，但 C 写的 main.c 调 jhyy 写的 lexer→parser→sema，C 写的 codegen 接 AST
6. `regress.py 43/46 全过`

### 4.2 ir.jhyy 翻译要点

- C `IRBuf { Arena *arena; char *buf; size_t len, cap; int next_tmp, next_block; }` → jhyy 等价 `type IRBuf = struct { arena: *Arena, buf: [*]u8, len: i64, cap: i64, next_tmp: i32, next_block: i32 }`
- `ir_emit` 用变参 — jhyy 无变参，**v1.0 拆成多个固定参数版本**（`ir_emit1` / `ir_emit2` / ...）或用 `StringBuilder` + `sb_appendf` 替代
- `ir_emit_ret` / `ir_emit_jnz` / `ir_emit_phi` 保持单 emit 形式

### 4.3 codegen.jhyy 翻译要点

- AST → QBE IL 是 phase-2 中**翻译 / 验证比最差**的模块（C 版就有大量特殊处理）
- 关键 oracle：**给定 AST，jhyy 版 codegen 输出 QBE IL 跟 C 版 byte-equal** — 这是 sprint 4 主要验证手段
- sprint 4 末做"核心场景 diff"：选 5-10 个 .jhyy 测试，diff C 版 codegen 跟 jhyy 版 codegen 输出的 .il

**struct pass-by-value sret 处理**：
- v0.4 ABI 锁定：sret pointer 作为第一个隐藏参数
- v0.6 已支持
- 翻译时按 C 版的 sret emit 模式 1:1

### 4.4 QBE IL 写盘（**关键风险**）

按 memory `feedback_qbe_crlf_root_cause`：jhyy 端写 QBE IL 必须 `fopen("wb")`，否则 Windows MSVCRT 把 `\n` 转 `\r\n` 污染 IL。

```jhyy
// 错误 ❌
let fp = libc_fopen(path, "w" as [*]u8);

// 正确 ✅
let fp = libc_fopen(path, "wb" as [*]u8);
```

**sprint 4 第一周**审查所有 fopen 调用，确保都用了 "wb"（jhyy 端）。这个 bug v0.6 修过 C 端（main.c:85, 418），sprint 4 翻译到 jhyy 端必须重做。

### 4.5 风险

- **QBE IL 写盘 binary mode** — sprint 4 必做正确
- codegen 字符串拼接：StringBuilder 已用（sprint 1）
- 大模块：**codegen.c 1552 行（phase-2 最大单模块，比 parser.c 982 行还大 58%）**，sprint 4 拆 3 个子 commit（codegen_expr → codegen_stmt → codegen_decl/ty），每 commit 末跑 regress，翻译工作量 ~2500 行 jhyy
- **变异构 QBE IL diff**：C 版 codegen 输出格式 vs jhyy 版输出格式，byte-equal 难达成 — 降级为"主要场景 diff" + 接受微小差异（如临时变量编号、空白）

### 4.6 验证

1. `jhyy (C 版) 编译 src0/{ir,codegen}.jhyy` 全成功
2. ir 单元测试：emit 各指令 + 临时变量命名
3. codegen 单元测试：每个 AST 节点 → QBE IL（INT / BINARY / CALL / IF / LET / MATCH / ...）
4. **关键 oracle**: 5-10 个 .jhyy 测试，C 版 codegen 输出 .il 跟 jhyy 版输出 .il diff（**目标 byte-equal，可接受微小差异**）
5. **M4 局部**: C 编译器走完整前端+后端 — 输入 .jhyy 源 → 输出 QBE IL → QBE 工具 → .s → gcc → .exe；regress.py 43/46 全过

### 5.3 Stage 1 验证脚本

# compiler/tests/bootstrap/stage1.sh
set -e

JHYY_C=./compiler/build/bin/jhyy            # C 版 (sprint 0)
JHYY_0=./compiler/build/bin/jhyy_0          # 第一代自举
JHYY_1=./compiler/build/bin/jhyy_1          # 第二代自举
SRC=compiler/src0/main.jhyy

# Step 1: C 版编 src0/main.jhyy → jhyy_0
echo "=== Step 1: jhyy (C) 编译 jhyy_0 ==="

# Step 2: jhyy_0 编 src0/main.jhyy → jhyy_1
echo "=== Step 2: jhyy_0 编译 jhyy_1 ==="
$JHYY_0 compile $SRC -o compiler/build/bin/jhyy_1

# Step 3: jhyy_0 和 jhyy_1 对相同 .jhyy 测试产物的 .il byte-equal 验证
echo "=== Step 3: .il byte-equal 验证（对齐 phase-2 L1 计划） ==="
for test in compiler/tests/examples/*.jhyy; do
  $JHYY_0 build $test -o /tmp/test_0.il
  $JHYY_1 build $test -o /tmp/test_1.il
  if ! diff /tmp/test_0.il /tmp/test_1.il > /dev/null; then
    echo "FAIL: $test"
    exit 1
  fi
done

# Step 4: regress.py 用 jhyy_1 跑（行为覆盖，不是字节）
echo "=== Step 4: regress.py 用 jhyy_1 ==="
PATH=./compiler/build/bin:$PATH python compiler/build/bin/regress.py
