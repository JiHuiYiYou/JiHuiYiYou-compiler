# v1.0.0 任务清单 + 概要设计

## Context

v0.6.0（自举前最后一期准备）通过，剩 3 个 P2/P3 已知限制（自举不依赖）。v0.6.1 patch 修了 v0.6 plan 文档，删了 jhyy-src/ 死代码。**v1.0.0 启动 phase-2 = 自举实施**。

按 2026-06-22 决策：
- **自举顺序**：先 phase-2（翻译 C 编译器前端到 jhyy）→ 后 v3.x+ QBE 完整重写
- **完成定义**：真自举闭环 = jhyy 编 jhyy 编 jhyy 跑 regress 全过（行为 diff，非字节）
- **完成时间**：估计 v1.5（main 翻译 + Stage 1 验证）

本文件用**粗粒度 5 sprint**划分（原 v0.6 § 下一版本 是细粒度 7 sprint，依赖链死板 + 无中间 oracle）。每 sprint 末做中间 oracle 验证（diff 翻译产物跟 C 版行为）。

## Sprint 1: 基础设施 (util + arena + diagnostics)

### 1.1 范围
- `compiler/src/util.c` (StringBuilder + 哈希表) → `compiler/jhyy-src/util.jhyy`
- `compiler/src/arena.c` (bump allocator, 95 行) → `compiler/jhyy-src/arena.jhyy`（v0.6 stage 0 翻译已删，需重新翻译）
- `compiler/src/diagnostics.c` (错误报告) → `compiler/jhyy-src/diagnostics.jhyy`

### 1.2 依赖
- 无（基础层）

### 1.3 验收
1. **编译验证**: jhyy 编译器（C 版）编译 jhyy-src/{util,arena,diagnostics}.jhyy
2. **行为验证**: util 测 StringBuilder + 哈希表；arena 跑 stage 0 6 项测试；diagnostics 测错误信息格式
3. **回归验证**: `regress.py 43/46 全过`
4. **M1 里程碑**: C 编译器**调 FFI 用 jhyy 写的 lib**（小范围：让某个 C 文件调 jhyy arena）

### 1.4 风险
- v0.6 翻译的 `compiler/jhyy-src/arena.jhyy` 在 2026-06-22 被删（与 examples/arena_test/ 重复 + arena_test.jhyy stale）。**sprint 1 重新翻译**
- jhyy 字符串 / 哈希表实现：jhyy 无 stdlib 字符串函数，自实现（用 `[*]u8` slice）
- FFI 兼容性：sprint 1 末首次让 C 编译器调 jhyy 写的 lib — C 编译器需新增 FFI 调用站点

## Sprint 2: 数据结构 (ast + types + symtab)

### 2.1 范围
- `compiler/src/ast.h/c` (AST 节点定义) → `compiler/jhyy-src/ast.jhyy`
- `compiler/src/types.h/c` (类型系统) → `compiler/jhyy-src/types.jhyy`
- `compiler/src/symtab.h/c` (符号表) → `compiler/jhyy-src/symtab.jhyy`

### 2.2 依赖
- Sprint 1 (util/arena/diagnostics)

### 2.3 验收
1. **编译验证**: jhyy 编 jhyy-src/{ast,types,symtab}.jhyy
2. **行为验证**: 单元测试 — 构造特定 AST/类型/符号表，验证 field / size / align / lookup
3. **回归验证**: regress.py 43/46
4. **里程碑**: C 编译器调 FFI 用 jhyy 写的 ast/types/symtab

### 2.4 风险
- AST 节点 tagged union 在 jhyy 实现：jhyy 无 enum + tagged union 原生支持，用 i32 kind + payload struct + as cast（v0.6 已验证）
- types.c 的递归类型（`Type *elem; Type **params`）在 jhyy 用 `*T` 实现 — arena 分配 + FFI 已有
- 哈希表在 sprint 1 util 已准备

## Sprint 3: 流水线 (lexer + parser + sema)

### 3.1 范围
- `compiler/src/lexer.h/c` → `compiler/jhyy-src/lexer.jhyy`
- `compiler/src/parser.h/c` (~900 行，phase-2 最大单模块) → `compiler/jhyy-src/parser.jhyy`
- `compiler/src/sema.h/c` → `compiler/jhyy-src/sema.jhyy`

### 3.2 依赖
- Sprint 1 + 2

### 3.3 验收
1. **编译验证**: jhyy 编 jhyy-src/{lexer,parser,sema}.jhyy
2. **行为验证**: 给定 .jhyy 源，jhyy 版 lexer 输出 token 序列跟 C 版 diff（最严 = byte-equal）；parser 走 parse_expr 等关键函数 diff
3. **回归验证**: regress.py 43/46
4. **里程碑**: C 编译器**走完整流水线**用 jhyy 写的 lexer/parser/sema（C 写的 main 调 jhyy 写的 lexer.jhyy → parser.jhyy → sema.jhyy → C 写的 codegen）

### 3.4 风险
- parser.c 900 行是 phase-2 最大单模块：sprint 3 拆 2-3 个子阶段（parse_decl → parse_stmt → parse_expr）
- lex/parse 状态机的 jhyy 实现：jhyy match 表达式功能够用（v0.6 已支持）
- sema 类型推断：i64 ↔ usize cast 已支持（v0.6 Sprint 6C）

## Sprint 4: 后端 (ir + codegen)

### 4.1 范围
- `compiler/src/ir.h/c` (IR 构建器) → `compiler/jhyy-src/ir.jhyy`
- `compiler/src/codegen.h/c` (AST → QBE IL) → `compiler/jhyy-src/codegen.jhyy`

### 4.2 依赖
- Sprint 1 + 2 + 3

### 4.3 验收
1. **编译验证**: jhyy 编 jhyy-src/{ir,codegen}.jhyy
2. **行为验证**: 给定 AST，jhyy 版 codegen 输出 QBE IL 跟 C 版 byte-equal（**关键 oracle** — codegen 输出是文本，比 lexer 简单）
3. **回归验证**: regress.py 43/46
4. **里程碑**: C 编译器走完整前端+后端 — 输入 .jhyy 源 → 输出 QBE IL → QBE 工具 → .s → gcc → .exe

### 4.4 风险
- QBE IL 写盘 `fopen("wb")` 必须在 jhyy 端重做（**v0.6 memory feedback** — Windows fopen "w" 把 \n 转 \r\n 污染 IL）
- codegen 大量字符串拼接（StringBuilder，sprint 1 已准备）
- struct pass-by-value sret 处理（v0.4 ABI 锁定）

## Sprint 5: main + Stage 1 验证

### 5.1 范围
- `compiler/src/main.c` (CLI 入口) → `compiler/jhyy-src/main.jhyy`
- 集成测试：jhyy（C 版）编 jhyy-src/main.jhyy → jhyy_0；jhyy_0 编 jhyy-src/main.jhyy → jhyy_1；jhyy_0 和 jhyy_1 对相同 .jhyy 源产生相同 .exe（行为 diff，非字节）

### 5.2 依赖
- Sprint 1 + 2 + 3 + 4

### 5.3 验收
1. **编译验证**: jhyy 编 jhyy-src/main.jhyy
2. **Stage 1 验证**: jhyy_0 编 jhyy-src/main.jhyy 得 jhyy_1；jhyy_0 和 jhyy_1 对相同 .jhyy 源产生相同 .exe（行为 diff）
3. **回归验证**: regress.py 43/46 用 jhyy_1 跑（**阶段闭环关键测试**）
4. **M4 里程碑**: **真自举闭环** = jhyy 编 jhyy 编 jhyy 跑 regress 全过

### 5.4 风险
- main.c 含 fopen/system/qbe 调用 + linker 路径，FFI 声明要全
- byte-equal vs 行为 diff：QBE IL byte-equal 易做到，.exe 字节不一定（QBE 内部 timestamp） — Stage 1 验证定义为"行为 diff"
- 首次 jhyy_0 可能很慢（jhyy 编译 jhyy 自己），需要耐心

## 风险总表

| 风险 | 缓解 |
|---|---|
| parser.c 900 行太大 | sprint 3 拆 2-3 个 commit（parse_decl → parse_stmt → parse_expr） |
| AST tagged union 在 jhyy 实现 | i32 kind + payload struct + as cast（v0.6 已验证） |
| 字符串 / 哈希表 jhyy 自实现 | sprint 1 util.jhyy 准备 |
| QBE IL 写盘 binary mode | sprint 4 codegen.jhyy 端用 fopen "wb"（v0.6 memory feedback） |
| jhyy 编译自己慢 | v2.x 优化期；v1.0 不优化 |
| FFI 兼容性（C 调 jhyy vs jhyy 调 C） | sprint 1 末做小范围试调，验证双向 FFI |
| 自研 OS 目标架构未定 | 当前 amd64_win 中间态，多目标设计 v3.x+ 考虑（见 memory `project_self_hosted_os_goal`） |

## 实施顺序

```
Sprint 1: util + arena + diagnostics         → M1 (jhyy lib 给 C 编译器用)
Sprint 2: ast + types + symtab                (全部 jhyy 化)
Sprint 3: lexer + parser + sema               (编译器核心 jhyy 化)
Sprint 4: ir + codegen                        (jhyy 写的后端)
Sprint 5: main + Stage 1 验证                 → M4 (真自举闭环)
```
