# v1.0.0 任务清单 + 概要设计

## Context

v0.6.0（自举前最后一期准备）通过，剩 3 个 P2/P3 已知限制（自举不依赖）。v0.6.1 patch 修了 v0.6 plan 文档，删了 jhyy-src/ 死代码。**v1.0.0 启动 phase-2 = 自举实施**。

按 2026-06-22 决策：
- **自举顺序**：先 phase-2（翻译 C 编译器前端到 jhyy）→ 后 v3.x+ QBE 完整重写
- **完成定义**：真自举闭环 = jhyy 编 jhyy 编 jhyy 跑 regress 全过 + **.il byte-equal**（对齐 phase-2 L1 计划 Step 2.5）
- **完成时间**：估计 v1.5（main 翻译 + Stage 1 验证）

本文件用**粗粒度 5 sprint**划分（原 v0.6 § 下一版本 是细粒度 7 sprint，依赖链死板 + 无中间 oracle）。每 sprint 末做中间 oracle 验证（diff 翻译产物跟 C 版行为）。

| Sprint | 范围 | 里程碑 | Est. Lines (jhyy) |

## Sprint 1: 基础设施 (arena + ffi + util)

### 1.1 范围
- `compiler/src/arena.c` (bump allocator, 94 行) → `compiler/src0/arena.jhyy`（v0.6 stage 0 翻译已删，从 `compiler/src/arena.c` 重新翻译）
- `compiler/src0/ffi.jhyy`（**新文件**，对齐 phase-2 L1 计划 Step 2.3）：列 libc 必需函数（malloc, free, memcpy, memset, fprintf, fopen, fread, fclose, strlen, strcmp, system, exit）
- `compiler/src0/util.jhyy`（**新文件**，**抽取型**）：
  - StringBuilder（从 codegen.c 13 处 sprintf/snprintf 调用抽 API）
  - HashMap（symtab.c 已有实现抽象）
  - Diagnostic（sema.c 21 处 error 调用抽 API）

### 1.2 依赖
- 无（基础层）

### 1.3 验收
1. **编译验证**: jhyy 编译器（C 版）编译 src0/{arena,ffi,util}.jhyy
2. **行为验证**: arena 跑 v0.6 stage 0 6 项测试；util 测 StringBuilder append 1000 次 + HashMap put/get 1000 项；ffi 测各 libc 声明链接通过
3. **回归验证**: `regress.py` 全过
4. **M1 里程碑**: C 编译器**调 FFI 用 jhyy 写的 lib**（小范围：让某个 C 文件调 jhyy arena + util StringBuilder）

### 1.4 风险
- **util.jhyy 是抽取型，不是翻译型**——sprint 1 末 codegen.c / symtab.c / sema.c 的相关代码**不重写**，只把 StringBuilder/HashMap/Diagnostic 的 API 抽象出来到 util.jhyy，供后续 sprint 使用。C 端的 sprintf/手写哈希表代码保留到对应 sprint 翻译时再删
- v0.6 翻译的 `compiler/src0/arena.jhyy` 在 2026-06-22 被删（与 examples/arena_test/ 重复 + arena_test.jhyy stale）。**sprint 1 重新翻译**
- jhyy 字符串 / 哈希表实现：jhyy 无 stdlib 字符串函数，自实现（用 `[*]u8` slice）
- FFI 兼容性：sprint 1 末首次让 C 编译器调 jhyy 写的 lib — C 编译器需新增 FFI 调用站点
- 变参 FFI（fprintf `...`）未做：v0.6 状态未确认，sprint 1 第 1 天**先验证**能否 extern fn 可变参；不行则用 sprintf → 字符串 → fputs 两步走

## Sprint 4: 后端 (ir + codegen)

### 4.1 范围
- `compiler/src/ir.h/c` (IR 构建器, 282 行) → `compiler/src0/ir.jhyy`
- `compiler/src/codegen.h/c` (AST → QBE IL, **1552 行 — phase-2 最大单模块**) → `compiler/src0/codegen.jhyy`

### 4.2 依赖
- Sprint 1 + 2 + 3

### 4.3 验收
1. **编译验证**: jhyy 编 src0/{ir,codegen}.jhyy
2. **行为验证**: 给定 AST，jhyy 版 codegen 输出 QBE IL 跟 C 版 byte-equal（**关键 oracle** — codegen 输出是文本，比 lexer 简单）
3. **回归验证**: regress.py 全过
4. **里程碑**: C 编译器走完整前端+后端 — 输入 .jhyy 源 → 输出 QBE IL → QBE 工具 → .s → gcc → .exe

### 4.4 风险
- **codegen.c 1552 行 = phase-2 最大单模块**（比 parser.c 982 行还大 58%）。**sprint 4 拆 3 子 commit**：codegen_expr → codegen_stmt → codegen_decl/ty，每 commit 末跑 regress
- QBE IL 写盘 `fopen("wb")` 必须在 jhyy 端重做（**v0.6 memory feedback** — Windows fopen "w" 把 \n 转 \r\n 污染 IL）
- codegen 大量字符串拼接（StringBuilder，sprint 1 已准备）
- struct pass-by-value sret 处理（v0.4 ABI 锁定）

## Sprint 5: main + Stage 1 验证

### 5.1 范围
- `compiler/src/main.c` (CLI 入口, 523 行) → `compiler/src0/main.jhyy`
- 集成测试：jhyy（C 版）编 src0/main.jhyy → jhyy_0；jhyy_0 编 src0/main.jhyy → jhyy_1；jhyy_0 和 jhyy_1 对相同 .jhyy 源产生 **byte-equal .il 输出**

### 5.2 依赖
- Sprint 1 + 2 + 3 + 4

### 5.3 验收
1. **编译验证**: jhyy 编 src0/main.jhyy
2. **Stage 1 验证**: jhyy_0 编 src0/main.jhyy 得 jhyy_1；jhyy_0 和 jhyy_1 对相同 .jhyy 测试集产生 **byte-equal .il**（diff 全空，对齐 phase-2 L1 计划 Step 2.5）
3. **回归验证**: regress.py 用 jhyy_1 跑（**阶段闭环关键测试**）
4. **M4 里程碑**: **真自举闭环** = jhyy 编 jhyy 编 jhyy 跑 regress 全过

### 5.4 风险
- main.c 含 fopen/system/qbe 调用 + linker 路径，FFI 声明要全
- **完成定义是 .il byte-equal，不是 .exe 字节、也不是 .exe 行为**：QBE IL 是纯文本输出，jhyy_0/jhyy_1 必须产 byte-identical 文件（参考 phase-2-self-hosting.md Step 2.5 verify.sh 脚本结构）。.exe 不验证字节（gcc 嵌 timestamp），行为由 regress.py 跑 jhyy_1 覆盖
- 首次 jhyy_0 可能很慢（jhyy 编译 jhyy 自己），需要耐心

| # | 标准 | 说明 |
|---|---|---|

## 风险总表

| 风险 | 缓解 |
|---|---|
| codegen.c 1552 行最大单模块 | sprint 4 拆 3 个 commit（codegen_expr → codegen_stmt → codegen_decl/ty） |
| parser.c 982 行太大 | sprint 3 拆 2-3 个 commit（parse_decl → parse_stmt → parse_expr） |
| AST tagged union 在 jhyy 实现 | i32 kind + payload struct + as cast（v0.6 已验证） |
| 字符串 / 哈希表 jhyy 自实现 | sprint 1 util.jhyy 抽取准备（从 codegen/symtab/sema 抽 API，C 端代码暂留） |
| QBE IL 写盘 binary mode | sprint 4 codegen.jhyy 端用 fopen "wb"（v0.6 memory feedback） |
| jhyy 编译自己慢 | v2.x 优化期；v1.0 不优化 |
| FFI 兼容性（C 调 jhyy vs jhyy 调 C） | sprint 1 末做小范围试调，验证双向 FFI |
| 变参 FFI（fprintf `...`）未验证 | sprint 1 第 1 天先验证 extern fn 能否 variadic；不行用 sprintf → fputs 两步 |
| 自研 OS 目标架构未定 | 当前 amd64_win 中间态，多目标设计 v3.x+ 考虑（见 memory `project_self_hosted_os_goal`） |

## 实施顺序

```
Sprint 1: arena + ffi + util(抽取)         → M1 (jhyy lib 给 C 编译器用)
Sprint 2: ast + types + symtab                (全部 jhyy 化)
Sprint 3: lexer + parser + sema               (编译器核心 jhyy 化)
Sprint 4: ir + codegen(拆 3 commit)           (jhyy 写的后端)
Sprint 5: main + Stage 1 byte-equal 验证     → M4 (真自举闭环)
```
