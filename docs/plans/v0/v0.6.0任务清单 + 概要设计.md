## Context

**v0.6.0 的定位**：
- ✅ 补齐"自举必须"的语言特性（切片 codegen、模块命名空间、`*T ↔ usize` 转换）

- ✅ Stage 0 试点：把 `arena.jhyy` 翻译出来，用 v0.6 编译器编译并对比 C 编译器输出
- ❌ **不**引入 v1.x 才需要的高级特性（闭包、泛型、async、function pointers）
- ❌ **不**做完整 Phase 2 翻译（那是 v1.0.0 / v1.1.0 的事）

### 为何 v0.6 → v1.0（不写 v0.7 / v0.8）

按 `docs/plans/phase/phase-1-v0.2.x-to-v1.0.0.md` 的"严格按依赖顺序推进"原则：
- v0.5 已经通过质量门禁

- v1.0 = 启动自举，**版本号跳跃标志 Phase 切换**（不再"准备"而是"做"）
- 后续 v1.1 / v1.2 = 逐步翻译 C 编译器模块到 JHYY

## Sprint 6A: 切片 codegen

**目标**：`[*]T` 在 sema 已接受，codegen 缺失。补齐后 parser 可以用 `[Token]` range，C 编译器源码中所有动态指针数组 (`Node **args` + `size_t nargs`) 可替换为 `[*Node]` slice。

### 6A.1 类型表示

`[*]T` 在 codegen 层 = `{ptr: *T, len: i32}` 24 字节布局：

```
offset  field
0       ptr  (*T)
8       len  (i64)
```

ABI：按 struct pass-by-value 走 sret。

### 6A.2 切片字面量

```jhyy
let arr: [i32; 3] = [1, 2, 3];
let s: [*i32] = arr as [*i32];   // 数组 → 切片（隐式 decay）
let s2: [*u8] = "hello" as [*u8]; // 字符串 → 字节切片
```

**实现**：codegen 把 array 字面量的栈地址 + 长度构造成 `{ptr, len}` 结构。

### 6A.3 索引与切片

```jhyy
let s: [*i32] = &[1, 2, 3];
let first = s[0];        // *(s.ptr + 0) — 已支持数组索引，扩展到切片
let slice = s[1..3];     // [*i32] 子切片 = {s.ptr+1, 2}
```

### 6A.4 `len(s)` 内置

```jhyy
fn len(s: [*i32]) -> i64 {
    s.len
}
```

**实现**：新增 `NODE_LEN` AST 节点 / `len` 内置函数。sema 接收 `[*]T` 或 `[T; N]`，codegen 返回 `.len` 字段。

### 6A.5 范围 for 迭代切片

```jhyy
for i in 0..len(s) {
    printf("%d ", s[i]);
}
```

(已是 for 循环范围，未变)

### 6A.6 回归测试

- `slice_basic.jhyy` — 切片字面量 + index + len
- `slice_subrange.jhyy` — `s[1..3]` 子切片
- `slice_from_array.jhyy` — 数组 → 切片 decay
- `slice_iterate.jhyy` — 用 for 遍历打印

## Sprint 6B: 模块命名空间

**目标**：`import math;` 后 `math::factorial(5)` 是唯一访问方式（不再扁平化为裸 `factorial`），避免多文件模块系统的名字冲突。

### 6B.1 符号 mangling

| 当前 (v0.4-v0.5) | v0.6 后 |
|------|------|
| `math.jhyy::factorial` 编译为 `$factorial` | `$math__factorial` (双下划线分隔) |
| 调用 `factorial(5)` 走 `$factorial` | 调用 `math::factorial(5)` 走 `$math__factorial` |

### 6B.2 语法

```jhyy
import math;

fn main_jhyy() -> i32 {
    math::factorial(5)        // 限定调用
    // factorial(5)           // ❌ 错误：未在当前模块定义
}
```

C ABI 兼容：mangled 名字仅在 JHYY 内部使用，extern fn 不受影响（`extern fn printf(...)` 仍编译为 `printf`，无 mangling）。

### 6B.3 实现细节

- `symtab.c` `Sym` 加 `module: *Sym` 字段，记录所属模块
- `sema.c` 解析 `expr::ident` 时，把 `expr` 当作 module 名查 symtab，找 `ident` 在该 module 下的定义
- `codegen.c` 发射函数调用时，按 module 前缀 mangle 名字
- `main.c` `compile` 合并多文件 decls 时保留 module 归属

### 6B.4 回归测试

- `multi_file_namespace.jhyy` — main.jhyy `import math; math::factorial(5)`
- 同名不同模块：`import lib_a; import lib_b; lib_a::process(); lib_b::process();`

### 6B.5 破坏性变更

**BREAKING**：所有 v0.5 之前的 `import` 用法变非法。需更新现有 v0.4 multi-file 测试：
- `multi_file/main.jhyy` 改为 `import math; math::factorial(5)`
- `mcp-jhyy` 的 `jhyy://examples` 资源生成的链接说明更新

## Sprint 6C: `as` 类型转换补全

**目标**：v0.5 已支持整数 ↔ 浮点 ↔ 浮点。v0.6 加 `*T ↔ usize` 和 `i64 ↔ usize` 用于 hash 计算和 ptr arithmetic。

### 6C.1 新增转换

| 从 → 到 | 实现 |
|---------|------|
| `*T → usize` | QBE 原生 ptr → int (零扩展) |
| `usize → *T` | QBE 原生 int → ptr |
| `i32 ↔ usize` | 隐式（编译期同尺寸语义） |
| `i64 → usize` | 当 i64 = usize 时直接 bitcast |
| `usize → i64` | 同上 |

### 6C.2 使用场景

```jhyy
let p: *u8 = &x;
let addr: usize = p as usize;       // ptr → int
let back: *u8 = addr as *u8;       // int → ptr
let hash: u64 = addr * 1099511628211; // FNV-1a 常量
```

### 6C.3 回归测试

- `ptr_usize_cast.jhyy` — ptr → usize → ptr round-trip
- `fnv_hash.jhyy` — 用 `as usize` 做 hash 计算

## Sprint 6D: Pre-existing bug 修复

回归目前 34/36 通过，2 个 pre-existing 失败必须在 v0.6 修掉（自举不能有"已知 failure"）。

### 6D.1 `import_test.jhyy` 路径 bug

**症状**：`import_test.jhyy` 找不到同目录 `mylib.jhyy`（错误: `cannot open 'compiler/tests/mylib.jhyy'`）。
**根因**：`main.c` 的 `resolve_imports` 用 `main_path` 的目录做 import 解析，但 `import_test.jhyy` 可能在不同 cwd 下被调用，cwd 与 main_path 目录不一致。
**修复**：`resolve_imports` 优先用 `main_path` 所在目录；找不到时 fallback 到 cwd。

### 6D.2 `mylib.jhyy` link bug

**症状**：`mylib.jhyy` 单独编译报 `collect2: error: ld returned 5 exit status` (gcc link 失败)。
**根因**：`main.c` `cmd_compile` 把 `mylib.jhyy` 当主模块编译，但 `mylib.jhyy` 没有 `main_jhyy` 函数，链接 runtime.c 时找不到入口点。
**修复**：`cmd_compile` 检测到没有 `main_jhyy` 时跳过链接，仅生成 `.il`（类似 `cmd_build`）。或：明确要求多文件编译时第一个文件必须是含 `main_jhyy` 的。

### 6E.2 翻译产物

`compiler/jhyy-src/arena.jhyy`：
```jhyy
// arena.jhyy — Stage 0 自举验证
// 等价于 compiler/src/arena.c

extern fn malloc(size: i64) -> *u8;
extern fn free(p: *u8);

type Arena = struct {
    base: *u8,
    cursor: *u8,
    end: *u8,
};

fn arena_init(a: *Arena, size: i64) {
    a.base = malloc(size);
    a.cursor = a.base;
    a.end = a.base + size;
}

fn arena_alloc(a: *Arena, size: i64) -> *u8 {
    // align to 8
    let aligned = (a.cursor + 7) & (-8 as i64) as *u8;
    a.cursor = aligned + size;
    aligned
}

fn arena_calloc(a: *Arena, size: i64) -> *u8 {
    let p = arena_alloc(a, size);
    // memset(p, 0, size) — extern fn memset
    extern fn memset(p: *u8, val: i32, n: i64) -> *u8;
    memset(p, 0, size);
    p
}

fn arena_free(a: *Arena) {
    free(a.base);
}
```

### 6E.3 验证

2. **运行**：`arena.exe` 单元测试通过（alloc 多 size + free 后 base == NULL 检查）
3. **对比**：与 C 编译器生成的 `arena.exe` 行为一致（分配相同 size 后 cursor 偏移相同）

## Risk Notes

### 6A (切片)

1. **sret 24 字节布局**：`[*]T` = `{*T, i64}` 共 16 字节，不是 24 字节——按实际写代码。Windows x64 struct 分类 ≤16 字节走寄存器，超出走 sret。需要确认 codegen 路径一致。
2. **数组 decay 语义**：`arr as [*T]` 是隐式 decay 还是显式 cast？v0.6 决策为**显式 cast**（不引入隐式转换，避免歧义）。

### 6B (命名空间)

1. **mangling 与 C 互操作**：extern fn 不 mangle，但 extern fn 不能"在 module 里"——module 里定义 extern fn 时相当于全局声明，调用时直接用原名。
2. **同名 symbol in different modules**：`lib_a::process()` 和 `lib_b::process()` 必须能共存——symtab 在 module scope 内不冲突，全局 scope 用 mangled name 区分。
3. **破坏性变更**：v0.5 multi-file 测试需要更新。

### 6C (`as`)

1. **指针 ↔ usize 语义**：零扩展（zext） vs 符号扩展（sext）？决策为**零扩展**（指针永远是非负）。
2. **i64 → usize 安全性**：i64 = usize 在 x64 上是同 size，但语义不同。i64 是数字，usize 是地址。决策为**显式 cast**（不隐式）。

### 6D (bug fix)

1. **import_test 路径 fallback**：fallback 到 cwd 可能在不同项目里行为不一致。先 fallback 到 main_path 目录，找不到时 fallback 到 cwd，最后失败报错。

### 6E (Stage 0)

1. **memset FFI**：arena_calloc 用到 memset，需要 FFI 声明 `extern fn memset(p: *u8, val: i32, n: i64) -> *u8`。memset 是 C 标准库，所有平台都有。
2. **alignment 算法**：C 用 `& ~(ALIGN-1)`，JHYY 没位运算 AND-not 简写。需展开成两步：
   ```jhyy
   let mask: i64 = -8;
   let aligned = (a.cursor as i64 + 7) & mask as *u8;
   ```
   （`as *u8` 把整数转回指针）
3. **Stage 0 ≠ Stage 1**：Stage 0 只是"JHYY 源 → C 编译器编译 → 二进制"，不验证自举产物能编译自身。

## 不在 v0.6 范围（明确延后）

| 特性 | 延后到 | 理由 |
|------|--------|------|
| C ABI struct 传递 (Windows x64 native classification) | v0.7+ | 自举不依赖（编译器只调 scalar C 函数） |
| Variadic FFI (`printf(...)`) | v0.7+ | 自举不用 printf (用 `extern fn puts`) |
| Function pointers / 闭包 | v1.x | Pratt parser 用 `match` 重写，无需 function pointer |
| 泛型 | v2.x | C 编译器无泛型 |
| async / await | v2.x | C 编译器同步 |
| fmod (`%` on f32/f64) | v0.7+ | 自举不用浮点 mod |
