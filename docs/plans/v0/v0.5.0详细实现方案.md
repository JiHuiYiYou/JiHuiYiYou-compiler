## Sprint 5A: 全部 Bug 修复

### 5A.1 修复 `*ptr = *ptr - expr` sema 类型推断

#### Problem
在 `sema.c` 的二元运算赋值 `*ptr = *ptr - expr` 中，RHS 上 `*ptr` 的类型推断走到 `infer_type` 处理 `NODE_BINARY` 时，递归调用的子节点类型推断路径在赋值场景下可能使用 LHS 的"待赋值引用"语义，导致 RHS 重复求值或类型不一致。

#### Root Cause
`sema.c` 中 `infer_type` 处理 `NODE_ASSIGN` 时，对 RHS 子表达式做 type inference 的过程中，把 `*ptr` 在 RHS 上下文里识别成了"被赋值的 LHS"，触发了不同的语义路径。

#### Files: `sema.c`

**Fix**: 在 `infer_type` 对 `NODE_ASSIGN` / `NODE_BINARY` 的处理中，确保 RHS 子表达式走完整的 type inference（包括 `NODE_DEREF` 路径），不应用 LHS 的赋值特殊语义。具体改动:
- `sema_infer_type(Node *n)` 中 `NODE_ASSIGN` 分支：先 `infer_type(n->assign.value)`，再校验 `infer_type(n->assign.target)` 与 RHS 类型一致。
- `NODE_BINARY` 分支：两侧独立 infer，互不影响。

#### Regression Test
`compiler/tests/examples/ptr_self_assign.jhyy`:
```
fn main_jhyy() -> i32 {
    let mut x: i32 = 100;
    let p: *mut i32 = &mut x;
    *p = *p - 30;
    *p
}
```
预期 EXIT: 70。

### 5A.2 修复嵌套 if/else if/else phi 前驱块标签

#### Problem
`codegen.c` 的 `cg_if` 在处理 `if a { ... } else if b { ... } else { ... }` 链时，构造 phi 节点的前驱块标签指向了错误的"中间块"。例如 phi 节点有两个前驱（cond_true / cond_false），但 emit 出来的标签都指向了 fallthrough 块。

#### Root Cause
原代码复用了"上一级 if 的 end 块"作为"下一级 if 的条件判断块"，但 phi 节点还在用更早的 end 块作为前驱。在三级嵌套 `if/else if/else` 时，这个错误会累加到第三级。

#### Files: `codegen.c`

**Fix**: 引入 trampoline 块策略 — 每个 else-if 都创建独立的 cond/end 块，最终统一汇入 join 块：

```
@if1_cond
    cbr %cond1, @if1_then, @if1_else

@if1_then
    <then body>
    jmp @join

@if1_else    (this is also @if2_cond)
    cbr %cond2, @if2_then, @if3_cond

@if2_then
    <else-if body>
    jmp @join

@if3_cond
    cbr %cond3, @if3_then, @if3_else
...
```

**关键改动**: phi 节点的 predecessors 必须是从各分支最后一条 `jmp` 之前那个块的标签，而不是 fallthrough 块。审计 codegen.c 中所有 emit_phi 调用，确保 `pred_labels[]` 与实际跳转路径匹配。

#### Regression Test
`compiler/tests/examples/nested_if.jhyy`:
```
fn main_jhyy() -> i32 {
    let x: i32 = 5;
    let result: i32 = 0;
    if x == 1 {
        result = 100;
    } else if x == 2 {
        result = 200;
    } else if x == 3 {
        result = 300;
    } else if x == 4 {
        result = 400;
    } else if x == 5 {
        result = 500;
    } else {
        result = 999;
    }
    result
}
```
预期 EXIT: 500。

### 5A.3 修复 if/else void 分支 phi 类型错误

#### Problem
void 返回函数里 if/else 链在 else 分支没有被实际进入时，phi 节点被填充成 `%t0` (默认 word 类型)，QBE 报错或运行期 crash。

#### Files: `codegen.c`

**Fix**: `cg_emit_phi` 增加 void 类型判断：

```c
if (type == VOID_TYPE) {
    // phi for void: only need to record control flow, no value
    ir_emit(ir, "    %%t%d =l phi @block1_pred, @block2_pred\n", phi.id);
} else {
    // existing scalar phi
}
```

实际方案：让 void 分支不参与 phi — 在 join 块起始处直接 fallback 到第一个非 void 分支的值。

#### Regression Test
`compiler/tests/examples/void_if.jhyy`:
```
fn do_nothing() -> () {
    let x = 1;
    if x == 1 {
        printf("yes\n");
    } else {
        printf("no\n");
    }
}

fn main_jhyy() -> i32 {
    do_nothing();
    0
}
```
预期 stdout: "yes\n", EXIT: 0。

### 5A.4 浮点算术 codegen (f32/f64)

#### Problem
`codegen.c` 的 `cg_binary` 对所有二元运算都用整数 opcode (addw / subw / mulw / divw / ...)，浮点算术结果为 0。

#### Files: `codegen.c`

**Fix**: 根据 `KIND_FLOAT` / `KIND_DOUBLE` 类型分发：

```c
case NODE_BINARY: {
    Type *t = lhs.type;
    char qt = qbe_type_of(t);
    const char *op;
    switch (d->op) {
        case BIN_ADD: op = "add"; break;
        case BIN_SUB: op = "sub"; break;
        case BIN_MUL: op = "mul"; break;
        case BIN_DIV: op = "div"; break;
        ...
    }
    // emit: %tN =<qt> <op> %lhs, %rhs
    if (qt == 's' || qt == 'd') {
        ir_emit(ir, "    %%t%d =%c %s %%t%d, %%t%d\n",
                result.id, qt, op, lhs.id, rhs.id);
    } else {
        ir_emit(ir, "    %%t%d =%c %s %%t%d, %%t%d\n",
                result.id, qt, op, lhs.id, rhs.id);
    }
}
```

注意 QBE 的浮点比较用 `eqs` / `les` / `ges` / `lts` / `gts` (单精度) 或 `eqd` / `led` / ... (双精度)。

#### Regression Test
`compiler/tests/examples/float_arith.jhyy`:
```

```

### 5A.5 break/continue 支持

#### Files: `lexer.h/c`, `parser.h/c`, `ast.h`, `sema.c`, `codegen.c`

**Step 1**: `lexer.h/c` 新增 token：
```c
typedef enum {
    ...
    TK_BREAK,
    TK_CONTINUE,
    ...
} TokenKind;
```
识别 `break` / `continue` 关键字。

**Step 2**: `ast.h` 新增节点：
```c
NODE_BREAK,
NODE_CONTINUE,
```

**Step 3**: `parser.h/c` 解析：
```c
case TK_BREAK: {
    Node *node = ast_new_break(arena, loc);
    expect(lexer, TK_SEMI);
    return node;
}
case TK_CONTINUE: {
    Node *node = ast_new_continue(arena, loc);
    expect(lexer, TK_SEMI);
    return node;
}
```

**Step 4**: `sema.c` 校验作用域 — break/continue 必须在 while/for 体内：
```c
typedef struct {
    int loop_depth;  // sema check 时维护
} SemaContext;

case NODE_BREAK:
case NODE_CONTINUE:
    if (sema->loop_depth == 0) {
        // error: break/continue outside loop
    }
```

**Step 5**: `codegen.c` 跳转 — codegen 时维护当前 loop 的 start/end 块标签栈：
```c
typedef struct {
    int loop_label_stack[MAX_LOOP_DEPTH];
    int loop_depth;
} CGContext;

// 在 cg_while / cg_for 入口:
cg->loop_start_label = ir_new_label("loop_start");
cg->loop_end_label = ir_new_label("loop_end");
cg->loop_depth++;

// cg_stmt 处理 NODE_BREAK:
ir_emit_jmp(cg->ir, cg->loop_end_label);

// cg_stmt 处理 NODE_CONTINUE:
ir_emit_jmp(cg->ir, cg->loop_start_label);
```

while 循环 QBE 结构：
```
@loop_start
    cbr %cond, @loop_body, @loop_end
@loop_body
    <body>
    jmp @loop_start
@loop_end
    <after loop>
```

for 循环类似，但 condition 是 `(i < end)`，body 末尾 `i = i + 1` 后 jmp 回 loop_start。

#### Regression Test
`compiler/tests/examples/break_continue.jhyy`:
```
fn main_jhyy() -> i32 {
    let sum: i32 = 0;
    for i in 0..100 {
        if i == 10 {
            break;
        }
        if i % 2 == 0 {
            continue;
        }
        sum = sum + i;
    }
    sum
}
```
预期: sum = 1+3+5+7+9 = 25, EXIT: 25。

### 5A.6 大整数溢出行为定义

#### Files: `docs/lang-spec/jhyy-lang-spec-v0.5.0.md`

新增章节"整数溢出语义":
> 整数算术运算采用二进制补码环绕（wrap-around）语义，与 LLVM / GCC 默认行为一致。运行时不做溢出检查。溢出检测由上层应用负责。
> - u8: 255 + 1 = 0
> - i8: 127 + 1 = -128
> - u32: 0 - 1 = 4294967295
>
> 调试期可启用 `-fsanitize=integer-overflow` (P1，未来)。

#### Regression Test
`compiler/tests/examples/overflow.jhyy`:
```
fn main_jhyy() -> i32 {
    let mut x: u8 = 200;
    let mut y: u8 = 100;
    let z: u8 = x + y;  // 300 → 44
    let w: i8 = -128;
    let v: i8 = w - 1;  // -129 → 127
    (z as i32) + (v as i32)  // 44 + 127 = 171
}
```
预期 EXIT: 171。

## Sprint 5B: ABI 白皮书 v1.0.0

### 输出
`docs/abis/jhyy-abi-v1.0.0.md`

### 章节大纲

#### 1. 引言
- 文档目的：定义 JHYY v0.5.0+ 的二进制接口规范
- 适用范围：v0.5.0 ~ v1.0.0 期间所有 .jhyy 编译产物
- 平台：Windows x64 PE (当前唯一验证平台)
- 与上游 C ABI 的关系

#### 2. 类型布局

| 类型分类 | 具体类型 | 布局 | 大小 (bytes) | 对齐 (bytes) |
|---------|---------|------|-------------|-------------|
| 整数 | i8 / u8 | 8-bit | 1 | 1 |
| 整数 | i16 / u16 | 16-bit | 2 | 2 |
| 整数 | i32 / u32 | 32-bit | 4 | 4 |
| 整数 | i64 / u64 | 64-bit | 8 | 8 |
| 浮点 | f32 | IEEE-754 single | 4 | 4 |
| 浮点 | f64 | IEEE-754 double | 8 | 8 |
| 布尔 | bool | 8-bit (QBE 'b' 对齐) | 1 | 1 |
| 字符 | char | 32-bit (Unicode scalar) | 4 | 4 |
| 字符串字面量 | *u8 | 指向 .rodata 的指针 | 8 | 8 |
| 指针 | *T | QBE 'l' | 8 | 8 |
| 数组 | [T; N] | N × sizeof(T), aligned to T | N * sizeof(T) | alignof(T) |
| 结构体 | struct {...} | 字段顺序排列，每个字段按其对齐填充 | Σ align_padded(field) | max(alignof(fields)) |
| 枚举 | enum {...} | 4-byte tag + max(payload) word | 4 + sizeof(max_payload) | 8 |
| 切片 (规划中) | [*]T | (data_ptr, len) | 16 | 8 |

#### 3. sizeof / alignof 实现
通过 `types.c` 的 `type_size()` / `type_align()` 函数实现：
```c
size_t type_size(Type *t) {
    switch (t->kind) {
        case KIND_I32: case KIND_U32: case KIND_FLOAT: return 4;
        case KIND_I64: case KIND_U64: case KIND_DOUBLE: case KIND_PTR: return 8;
        case KIND_STRUCT: /* sum of padded fields */ ...
        ...
    }
}
```

#### 4. 函数调用约定
- 参数顺序：从左到右（QBE 原生）
- 参数传递：标量按值（寄存器或栈），struct 全部按值（调用方分配栈拷贝，传指针）
- 返回值：
  - 标量 (i8/i16/i32/i64/f32/f64/pointer)：单寄存器
  - struct (KIND_STRUCT)：sret — 调用方分配返回槽，传隐藏指针为第一个参数
- QBE 签名：
  - 标量返回: `function $name(w %p1, l %p2) -> { w, w }` 或简写 `function $name(w %p1, l %p2)`
  - struct 返回: `export function $name(l %ret, w %p1, l %p2)`
  - void 返回: `export function $name(w %p1, l %p2)`

#### 5. struct pass-by-value 详细规则

```
调用方 (caller):
  1. 计算参数 struct 类型大小 sz
  2. alloc sz 字节栈空间 copy_slot
  3. 逐字段 load src + store 到 copy_slot
  4. 传 copy_slot (qbe_type='l') 给被调用方

被调用方 (callee):
  1. 形参声明为 `l %name` (qbe_type='l', value IS 指针)
  2. 字段访问: src_ptr + offset → load
```

#### 6. QBE IL 指令映射速查表

| JHYY 表达式 | QBE IL |
|------------|--------|
| `let x: i32 = 42` | `%t0 =w copy 42` |
| `x + y` | `%t1 =w add %t0, %t2` |
| `x == y` | `%t1 =w ceq %t0, %t2` |
| `*p` | `%t1 =w loadw %p` |
| `&x` | `%t1 =l alloc 4` + `storew %x, %t1` |
| `arr[i]` | `%off =w mul %i, 4` → `%addr =l add %arr_base, %off` → `%v =w loadw %addr` |
| `struct.field` | `%addr =l add %struct_ptr, %offset` → `%v =w loadw %addr` |
| `if cond` | `jnz %cond, @then, @else` |
| `fn call` | `%t =w call $fn(...)` |
| `struct call` | `call $fn(l %ret_slot, ...)` |
| `return val` | `ret %val` 或 `ret` (void) |

#### 7. FFI 边界规则

- extern fn 名字不修饰 (C 原样)
- extern fn 参数和返回类型必须是 C ABI 兼容的 (i32/u32/i64/f32/f64/*u8)
- struct 不能直接传给 C 库 (P2 限制)
- 调用方使用 C 调用约定，Windows x64 PE
- printf 中文输出需要 SetConsoleOutputCP(CP_UTF8) (runtime.c 已处理)

#### 8. 名称修饰
- JHYY 内部符号：不修饰 (`main_jhyy`, `add_i32`)
- extern 符号：C 原样 (`printf`, `fopen`)
- 所有符号最终通过 `export function $name` 暴露给链接器

#### 9. 内存模型
- 栈：QBE `alloc` 指令分配，函数返回时自动释放
- 静态数据：QBE `data` 段 → PE `.rdata` 节 (字符串字面量)
- Arena allocator：FFI 提供 `arena_init` / `arena_alloc` / `arena_reset` / `arena_destroy`，编译器和用户程序共用

#### 10. Windows x64 PE 特定行为
- 入口点：runtime.c 的 `main()` 调用 `main_jhyy()`
- UTF-8 控制台输出：runtime 启动时 SetConsoleOutputCP(CP_UTF8)
- 路径分隔符：使用 `/`，QBE 和 GCC 在 Windows 上都接受
- QBE target: `-t amd64_win`
- GCC 链接: `gcc output.s runtime/runtime.c -o output.exe -lm`

#### 11. C ABI 互操作
- ✅ 直接兼容: `i32`, `u32`, `i64`, `u64`, `f32`, `f64`, `*u8`, `*T`
- ⚠️ 需要 adapter: bool (C 用 i32, JHYY 用 1 byte), char (C 是 8-bit, JHYY 是 32-bit Unicode)
- ❌ 不兼容: struct (需要手写 adapter), enum (需要手写 tag mapping)

#### 12. 版本兼容性承诺
- v0.5.0 ~ v1.0.0 期间保证 ABI 稳定
- 任何 ABI 改动必须走 v2.x 主版本号
- Lang spec 与 ABI 白皮书必须同步更新

            text=True,
            timeout=30,
            encoding='utf-8'

        return {
            "success": run_result.returncode == 0,
            "exit_code": run_result.returncode,
            "stdout": run_result.stdout,
            "stderr": run_result.stderr,
            "compile_diagnostics": compile_result["diagnostics"],
            "run_diagnostics": []
        }

# ========== 辅助函数 ==========

DIAG_RE = re.compile(r"^(.+?):(\d+):(\d+):\s*(error|warning):\s*(.+)$")

def _run_jhyy(cmd: list) -> dict:
    """运行 jhyy 命令并解析输出"""
    try:
        result = subprocess.run(
            cmd,
            capture_output=True,
            text=True,
            timeout=60,
            encoding='utf-8'
        )
        diagnostics = []
        for line in (result.stdout + result.stderr).splitlines():
            m = DIAG_RE.match(line)
            if m:
                diagnostics.append({
                    "file": m.group(1),
                    "line": int(m.group(2)),
                    "col": int(m.group(3)),
                    "severity": m.group(4),
                    "message": m.group(5),
                })
        return {
            "success": result.returncode == 0,
            "exit_code": result.returncode,
            "stdout": result.stdout,
            "stderr": result.stderr,
            "diagnostics": diagnostics,
        }
    except subprocess.TimeoutExpired:
        return {"success": False, "error": "Compilation timeout (60s)"}

## Potential Pitfalls

1. **break/continue 与 sret 函数**: break 在 sret 函数中不写 sret_slot — 这意味着提前 break 后 sret_slot 的值未初始化。需要确认调用方不会读取 sret_slot (一般不会，因为已经放弃返回值)。

2. **嵌套 if 与 break**: if 分支里有 break，codegen 必须正确处理"break 跳过 phi 合并块"。这需要把 break 视为 join 块的一个额外前驱。

3. **浮点比较的 QBE opcode**: QBE 比较指令末尾的 s/d/l/w 对应浮点/双精度/长/字，**与一般指令相反** — `eqs` 是单精度浮点等于，`eql` 是 64-bit 整数等于。codegen 必须按类型正确选后缀。

4. **MCP 服务 Windows 路径**: subprocess 调用 jhyy.exe 时，路径必须用 Windows 风格 (`C:\\...`) 或者用 forward slash (`C:/...`)。不能混用。

5. **MCP 服务中文输出**: Windows subprocess 默认 cp936 (GBK)，会破坏 UTF-8 输出。需要显式 `encoding='utf-8'` + `errors='replace'`。

6. **ABI 白皮书准确度**: 必须用 `offsetof!` 风格手算验证 — 写一个 C 测试程序，定义等价的 struct，对比 sizeof/offsetof 是否一致。

7. **big_test.jhyy 编译时间**: 300+ 行的 JHYY 程序编译时间应该在 5 秒以内。如果超时，需要排查。
